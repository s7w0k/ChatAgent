# 智扫通智能客服系统架构文档

## 一、项目概述

智扫通智能客服系统是一个基于大语言模型（LLM）的智能对话系统，专门为扫地机器人产品提供售前咨询和售后服务支持。系统采用现代化的 AI Agent 架构，结合 RAG（检索增强生成）技术，能够从专业知识库中检索相关信息，为用户提供准确、专业的回答。

该系统的主要功能包括：回答用户关于扫地机器人产品的各类问题、根据用户需求生成个性化的使用报告、主动收集用户反馈并处理投诉问题。系统通过直观的 Web 界面与用户进行交互，同时预留了 MCP（Model Context Protocol）接口，支持与其他 AI 系统的集成扩展。

### 2.2 分层架构说明

系统采用清晰的分层架构设计，每一层都有明确的职责边界：

**用户交互层**负责处理用户界面的展示和交互逻辑。Streamlit 框架提供了简洁的 Web 界面，用户可以输入问题并查看 AI 的回复。系统维护会话历史，支持多轮对话，让用户获得连贯的服务体验。

**Agent 核心层**是系统的大脑，负责协调各个组件的工作。ReactAgent 封装了 LangChain 的 Agent 能力，能够自主推理并调用合适的工具来完成任务。这一层还包含中间件机制，用于监控工具执行、记录日志、动态切换提示词等。

**RAG 服务层**实现了检索增强生成功能。当用户提出需要专业知识的问题时，系统会先从向量数据库中检索相关的参考资料，然后将这些信息连同用户问题一起发送给大语言模型，生成更加准确和有依据的回答。

**模型服务层**提供了统一的模型调用接口。系统使用阿里云 DashScope 的通义千问作为对话模型，使用 text-embedding-v4 作为文本嵌入模型。通过工厂模式设计，便于未来替换或升级底层模型。

**数据与配置层**存储系统运行所需的各种配置和数据。YAML 配置文件定义了模型参数、向量库配置、Agent 行为等；知识库文件夹存放待导入的 PDF 和 TXT 文档；向量数据库持久化存储嵌入向量。

## 三、技术栈详解

### 3.1 核心技术栈

| 技术类别 | 技术名称 | 版本要求 | 用途说明 |
|---------|---------|---------|---------|
| **Web 框架** | Streamlit | ≥1.35.0 | 构建用户交互界面 |
| **AI Agent 框架** | LangChain | ≥0.3.0 | Agent 构建与工具编排 |
| **图执行引擎** | LangGraph | 运行时依赖 | 中间件与状态管理 |
| **向量数据库** | Chroma | ≥0.1.0 | 知识向量化存储与检索 |
| **大语言模型** | 通义千问 (Qwen3-max) | - | 阿里云 DashScope API |
| **文本嵌入** | text-embedding-v4 | - | 文本向量化 |

### 3.2 LangChain 核心概念

LangChain 是一个用于构建 LLM 应用的开发框架，本项目主要使用了以下核心组件：

**create_agent** 是 LangChain 0.3.x 版本提供的 Agent 创建接口。与旧版本的 AgentExecutor 不同，create_agent 采用更加现代化的设计，支持流式输出、更灵活的中间件机制、更好的类型提示等特性。本项目的 ReactAgent 类使用 create_agent 将大语言模型与一组工具绑定，创建一个能够自主决策和执行任务的智能代理。

**Tool** 是 LangChain 中定义工具的标准方式。通过 @tool 装饰器，可以将任意 Python 函数封装为 Agent 可调用的工具。每个工具包含名称、描述和执行逻辑，Agent 会根据用户问题自动判断需要调用哪些工具。

**Middleware** 是 LangChain Agent 的扩展机制，类似于 Web 框架的中间件概念。本项目定义了三个中间件：monitor_tool 用于监控工具执行过程和结果；log_before_model 在模型调用前记录日志；report_prompt_switch 根据上下文动态切换提示词模板。这种设计使得非业务逻辑（如日志、监控）可以与核心业务逻辑解耦。


## 四、核心模块设计

### 4.1 ReactAgent 模块

ReactAgent 是整个系统的核心入口，封装了 LangChain Agent 的创建和调用逻辑。

```python
class ReactAgent:
    def __init__(self):
        self.agent = create_agent(
            model=chat_model,
            system_prompt=load_system_prompts(),
            tools=[rag_summarize, get_weather, ...],
            middleware=[monitor_tool, log_before_model, report_prompt_switch]
        )
```

该模块的主要职责包括：加载系统提示词模板、初始化工具集合、配置中间件、接收用户查询、调用 Agent 执行并返回流式响应。execute_stream 方法实现了流式输出，让用户能够实时看到 AI 的思考过程。

### 4.2 工具模块 (agent_tools)

系统定义了 7 个工具，涵盖了智能客服的典型场景：

| 工具名称 | 功能描述 | 参数 |
|---------|---------|------|
| rag_summarize | 从向量知识库检索专业资料 | query: str |
| get_weather | 查询指定城市天气 | city: str |
| get_user_location | 获取用户所在城市 | 无 |
| get_user_id | 获取用户 ID | 无 |
| get_current_month | 获取当前月份 | 无 |
| fetch_external_data | 获取用户使用记录 | user_id: str, month: str |
| fill_context_for_report | 触发报告生成上下文注入 | 无 |

这些工具使用 @tool 装饰器定义，LangChain 会自动提取函数签名和文档字符串，生成工具描述供 Agent 决策参考。

### 4.3 中间件模块 (middleware)

中间件机制是本项目的技术亮点之一，它实现了横切关注点与业务逻辑的分离：

**monitor_tool** 是一个工具执行监控器，使用 @wrap_tool_call 装饰器。它在工具执行前后进行日志记录，特别地，当检测到 fill_context_for_report 工具被调用时，会在运行时上下文（runtime.context）中设置 report 标志为 True。

**log_before_model** 是一个模型调用前置拦截器，使用 @before_model 装饰器。它在每次调用大语言模型之前输出日志，记录当前会话的消息数量和最后一条消息的内容，便于问题排查和对话分析。

**report_prompt_switch** 是一个动态提示词切换器，使用 @dynamic_prompt 装饰器。它检查运行时上下文中的 report 标志，如果为 True 则加载报告生成的专用提示词模板，否则加载通用的系统提示词。这种设计实现了"一套代码、多种场景"的灵活架构。

### 4.4 RAG 服务模块 (rag_service)

RAG（检索增强生成）是本项目实现专业问答的关键技术：

```python
class RagSummarizeService:
    def rag_summarize(self, query: str) -> str:
        # 1. 从向量库检索相关文档
        context_docs = self.retriever.invoke(query)
        
        # 2. 组装上下文
        context = ""
        for doc in context_docs:
            context += f"【参考资料】:{doc.page_content} | {doc.metadata}"
        
        # 3. 调用 LLM 生成答案
        return self.chain.invoke({"input": query, "context": context})
```

RAG 的工作流程是：用户问题首先通过嵌入模型转换为向量，然后在 Chroma 向量数据库中进行相似度搜索，找到最相关的 K 条文档（K=3，可配置），最后将检索到的文档内容作为上下文，连同用户问题一起发送给大语言模型，生成有据可查的回答。

### 4.5 向量存储模块 (vector_store)

VectorStoreService 负责向量数据库的管理：

**文档加载**：支持从指定文件夹批量加载 PDF 和 TXT 文件。系统通过 MD5 哈希值去重，避免重复导入已处理的文档。

**文本切分**：使用 RecursiveCharacterTextSplitter 将长文档切分为适合嵌入的文本块。默认配置为每块 200 字符，相邻块之间有 20 字符的重叠，保证语义连贯。

**向量检索**：提供 as_retriever 方法创建检索器，支持自定义返回结果数量（K 值）。

### 4.6 模型工厂模块 (factory)

ModelFactory 采用工厂模式封装模型初始化逻辑：

```python
class ChatModelFactory(BaseModelFactory):
    def generator(self) -> BaseChatModel:
        return ChatTongyi(model=rag_conf['chat_model_name'])

class EmbeddingsFactory(BaseModelFactory):
    def generator(self) -> Embeddings:
        return DashScopeEmbeddings(model=rag_conf['embedding_model_name'])
```

当前配置使用阿里云 DashScope 的通义千问模型：chat_model 为 qwen3-max（对话模型），embedding_model 为 text-embedding-v4（嵌入模型）。配置文件位于 config/rag.yml，修改配置即可切换模型。

## 五、数据流与执行流程

### 5.1 用户问答流程

```
用户输入问题
    │
    ▼
Streamlit 界面接收
    │
    ▼
ReactAgent.execute_stream()
    │
    ▼
┌──────────────────────────────────────────┐
│         LangChain Agent 推理循环          │
│                                          │
│  1. 分析用户问题                           │
│  2. 决定是否需要调用工具                    │
│  3. 如果需要，调用对应工具                   │
│  4. 工具返回结果                           │
│  5. 基于工具结果继续推理                    │
│  6. 生成最终回答                           │
└──────────────────────────────────────────┘
    │
    ▼
流式输出到界面
    │
    ▼
用户看到回答
```

### 5.2 报告生成流程

当用户请求生成使用报告时，系统会触发一系列工具调用：

```
用户："给我生成我的使用报告"
    │
    ▼
Agent 调用 get_user_id()
    │
    ▼
Agent 调用 get_current_month()
    │
    ▼
Agent 调用 fill_context_for_report()
    │  (middleware 捕获此调用，设置 context["report"] = True)
    │
    ▼
report_prompt_switch 中间件检测到 report=True
    │
    ▼
切换为报告生成专用提示词
    │
    ▼
Agent 调用 fetch_external_data(user_id, month)
    │
    ▼
Agent 整合所有信息生成完整报告
    │
    ▼
返回报告给用户
```

这个流程展示了 Agent 的自主决策能力和中间件的协调作用：Agent 不需要知道何时切换提示词，中间件会透明地完成这项工作。

### 5.3 RAG 问答流程

```
用户："小户型适合哪些扫地机器人？"
    │
    ▼
Agent 调用 rag_summarize("小户型 扫地机器人")
    │
    ▼
RagSummarizeService.retriever.invoke()
    │
    ▼
Chroma 向量库相似度搜索 (K=3)
    │
    ▼
返回 3 条最相关的文档片段
    │
    ▼
组装上下文：
【参考资料1】:... | 【参考资料2】:... | 【参考资料3】:...
    │
    ▼
调用 LLM (qwen3-max) 生成回答
    │
    ▼
返回带引用的专业回答
```

## 六、配置说明

### 6.1 配置文件结构

项目配置文件位于 config/ 目录，包含以下文件：

**rag.yml** - 大语言模型配置

```yaml
chat_model_name: qwen3-max      # 对话模型名称
embedding_model_name: text-embedding-v4  # 嵌入模型名称
```

**chroma.yml** - 向量数据库配置

```yaml
collection_name: agent          # 向量集合名称
persist_directory: chroma_db    # 数据库持久化路径
k: 3                            # 检索返回结果数量
data_path: data                 # 知识库文件路径
md5_hex_store: md5.text         # 文件哈希存储路径
allow_knowledge_file_type: ["pdf", "txt"]  # 支持的文件类型
chunk_size: 200                 # 文本块大小
chunk_overlap: 20               # 文本块重叠大小
separator: ["\n", "\n\n", ...]  # 文本分割符
```

**agent.yml** - Agent 行为配置

```yaml
external_data_path: data/external_data.csv  # 外部数据文件路径
```

### 6.2 提示词模板

提示词模板通过 config/prompts.yml 统一管理，支持动态加载。系统默认包含两类提示词：

**系统提示词**（system_prompt）：定义 Agent 的角色定位、行为规范、知识边界等。

**报告提示词**（report_prompt）：专门用于生成用户使用报告，包含报告结构、数据展示方式等专业指导。

中间件机制会根据上下文自动切换合适的提示词模板。
